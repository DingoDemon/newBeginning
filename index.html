发表于 2020-06-14 |

鸣谢:

深入理解 RecyclerView 的缓存机制

RecyclerView预加载机制源码分析

概述

RecyclerView 在滑动中不能每次都将移除的View销毁，创建新的添加进来。也不能无限地添加不销毁。需要使用复用机制，将移去的View缓存下来。移入时添加进来。 甚至在多个RecyclerView之间，也可以使用缓存。RecylerView抽离Recycler来进行缓存。

掌握Recycler缓存原理是进行RecyclerView性能优化的前提。

Recycler

A Recycler is responsible for managing scrapped or detached item views for reuse.

A “scrapped” view is a view that is still attached to its parent RecyclerView but that has been marked for removal or reuse

Typical use of a Recycler by a LayoutManager will be to obtain views for an adapter’s data set representing the data at a given position or item ID.
If the view to be reused is considered “dirty” the adapter will be asked to rebind it.
If not, the view can be quickly reused by the LayoutManager with no further work.

Biner(二)

发表于 2020-06-09 |

一些基础类的定义

IInterface

定义了Server具有哪些能力

IBinder

远程对象的接口，轻量级高性能跨进程调用的核心,远程对象的抽象协议。只要正确地实现了这个接口，这个对象就具备跨进程传输的能力。但是，不能实现这个接口。而是继承Binder类

Binder 和 BinderProxy

Binder是对于IBinder接口的标准实现。Java 层的 Binder 类，代表的其实就是 Binder 本地对象。BinderProxy 类是 Binder 类的一个内部类，它代表远程进程的 Binder 对象的本地代理；这两个类都实现了 IBinder, 因而都具有跨进程传输的能力；实际上，在跨越进程的时候，Binder 驱动会自动完成这两个对象的转换。

Stub

AIDL 的时候，编译工具会给我们生成一个名为 Stub 的静态内部类；这个类继承了 Binder, 说明它是一个 Binder 本地对象，它实现了 IInterface 接口，表明它具有 Server 承诺给 Client 的能力；Stub 是一个抽象类，具体的 IInterface 的相关实现需要开发者自己实现。

发表于 2020-05-28 | 分类于 Android |

View的三大流程概述

  • Measure
  • Layout
  • Draw

一个View经过测量，布局，绘制，三个流程之后才能展示在屏幕上。

  1. View从根布局开始,测量每一个子View，每一个子View又递归测量自身。并且把测量的值储存下来。
  2. 接下来，根布局对每一个子View进行布局，安排其摆放位置。子View将其储存起来，子Viw再递归对子View进行布局。
  3. 绘制自身，绘制子View，最终，将View展示在屏幕上。

整个过程由ViewRootImpl进行调度即（performTraversals()）。在此之前，需要先搞清楚根布局,ViewRootImpl是创建，并且和Activity 关联起来的。
