简述

Active resources are those that have been provided to at least one request and have not yet
been released. Once all consumers of a resource have released that resource, the resource then
goes to cache. If the resource is ever returned to a new consumer from cache, it is re-added to
the active resources. If the resource is evicted from the cache, its resources are recycled and
re-used if possible and the resource is discarded. There is no strict requirement that
consumers release their resources so active resources are held weakly.

Android 基于 Choreographer 的渲染机制详解

Android 怎么就不卡了呢之Choreographer

60帧的由来和垂直同步

60hz和60帧的区别

帧数是指运行时每秒所运行的帧数(简称FPS,Frames Per Second),帧数越大,画面越流畅,但是对资源消耗也就越大

hz指的是屏幕刷新的频率,比如屏幕刷新率是 75 hz,软件是 60 fps。每秒软件渲染60次,屏幕刷新 75 次,是没有啥效果的。除了重复帧率费电。

引入 Vsync 之前的 Android 版本,渲染一帧相关的Message,中间是没有间隔的,上一帧绘制完,下一帧的 Message 紧接着就开始被处理。这样的问题就是,帧率不稳定,可能高也可能低

因为人眼与大脑之间的协作无法感知超过60fps的画面更新,此时就需要用到稳定的60fps来达到想要的效果。

维持帧数围绕三部分展开,VSYNC,Triple Buffer,Choreographer

VSYNC && Triple Buffer

界面的显示大体会经过CPU的计算-> GPU合成栅格化->显示设备显示三个流程

VSYNC 每16.67ms发送一次,轮番展示画面ABC

之所以使用三级缓存的原因为最大化利用CPU,GPU。

例如:

  • 第一次VSYNC,屏幕展示画面A,CPU处理画面B,GPU等待CPU处理完毕

  • 第二次VSYNC,由于GPU在在处理画面B,所以画面停留在画面A,产生丢帧。但这时,CPU已经开始处理画面C了

  • 第三次VSYNC,GPU处理完了画面B,显示画面B,CPU处理画面A,GPU处理刚才第二次VSYNC中CPU处理的画面C。甚至在之后还能完成画面C继续处理画面A

  • 第四次VSYNC,展示画面C。CPU/GPU处理画面B,画面A已经早早准备好了,等待下次信号

假如使用两级缓存,显示器不停替换画面1和画面2的话。如果CPU或GPU之一处理超时。势必造成另一方的空闲等待。

Touch事件从触摸到Activity

  1. 当屏幕被触摸，Linux内核会将硬件产生的触摸事件包装为Event存到/dev/input/event[x]目录下。

  2. Input系统—InputReader线程：loop起来让EventHub调用getEvent()不断的从/dev/input/文件夹下读取输入事件。然后转换成EventEntry事件加入到InputDispatcher的mInboundQueue。

  3. Input系统—InputDispatcher线程：从mInboundQueue队列取出事件，转换成DispatchEntry事件加入到connection的outboundQueue队列。再然后开始处理分发事件 （比如分发到ViewRootImpl的WindowInputEventReceiver中），取出outbound队列，放入waitQueue.

  4. Input系统—UI线程：创建socket pair，分别位于”InputDispatcher”线程和focused窗口所在进程的UI主线程，可相互通信。

RecyclerView(二) - 缓存

发表于 2020-06-14

鸣谢:

深入理解 RecyclerView 的缓存机制

RecyclerView预加载机制源码分析

概述

RecyclerView 在滑动中不能每次都将移除的View销毁，创建新的添加进来。也不能无限地添加不销毁。RecyclerView内部实现了三级缓存，使之能够:

  • 上下滚动时利用复用机制，往下滑动时将移去的View缓存下来。往回滑动时如果可复用，移入时从缓存里添加回来。

  • 在向一个方向滑动时，计算即将滑入的View，并在规定时间内，尝试将其加入缓存。更高效地利用UIThread

  • 在全局刷新(NotifyDataSetChange)的时候，亦有缓存的应用,将还可复用的item，再加回来。

  • 甚至在多个RecyclerView之间，如果是相同的Adapter&&ViewType，那么无疑需要使用缓存。
