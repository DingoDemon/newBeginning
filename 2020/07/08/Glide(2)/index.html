Glide源码阅读笔记(二) - 缓存机制

发表于 2020-07-08

Glide 版本 4.11

以默认缓存策略追溯代码

简述

Active resources are those that have been provided to at least one request and have not yet
been released. Once all consumers of a resource have released that resource, the resource then
goes to cache. If the resource is ever returned to a new consumer from cache, it is re-added to
the active resources. If the resource is evicted from the cache, its resources are recycled and
re-used if possible and the resource is discarded. There is no strict requirement that
consumers release their resources so active resources are held weakly.

这是Engine里的一段注释,描述了ActiveResources的定义。我理解为仍在界面上已经被加载没有被回收的图片。

Glide记载过程中用到的缓存分为三级

  1. ActiveResources
  2. MemoryCache
  3. DiskCache

接下来逐一拆解Glide如何缓存加载图片

如果Target上已经有request的话,并且两者等价

等效的意思为同样的宽高，同样的model，同样的接收resourceClass,同样的优先级，和相同数量的Listener。这里不使用equals/hashCode的原因是因为不要求RequestListener实现equals/hashcode。

如果两者等价，则让之前的request去请求。

可以总结为:
加载同样的model到同一个target上，如果两次model和target都未改变。并且跳过内存缓存，则让之前的请求再去加载一次。

Engine # loadFromMemory

这一部分是从内存中获取缓存

可以看到内存缓存被细化成了两部分，一个是开头提到的ActiveResources,另一部分是MemoryCache

loadFromActiveResources

Map<Key, ResourceWeakReference> activeEngineResources = new HashMap<>()

从ActiveResources取EngineResource的逻辑为,HashMap中通过Key取得Resource的弱引用。
如果取到了的话，判断弱引用中对象是否已经被GC，如果GC了的话则从activeEngineResources中移出。
否则使其其引用计数acquired++。并返回

acquired用于资源recycle和release。执行release的时候，如果引用计数降到0的话，会调用Engine中的onResourceReleased。让其从ActiveResources中移除。执行recycler的时候，会让其内部资源，执行recycler。举个例子BitmapResource的recycler会bitmapPool.put(bitmap)

loadFromCache

MemoryCache接口的默认实现为LruResourceCache,继承自LruCache<Key, Resource<?>>。调用的即是LruCache的remove方法,即从LinkedHashMap中remove，这里不扩展。

流程很简单，从LruCache中remove掉Resource。如果成功，则将其激活。返回缓存。

LruResourceCache 默认大小探寻

可以得出LruResourceCache的缓存大小在大内存机器上为2屏幕图片*2的大小

DecodeJob # runWrapped

回到DecodeJob中。这一部分，是从磁盘中获取缓存

俯瞰整个流程

从头往后的Stage顺序

从前往后根据runReason转换DecodeJob.Stage，根据DecodeJob.Stage得到DataFetcherGenerator

DataFetcherGenerator实现有三个:

  • ResourceCacheGenerator,从cache中查找已经被转码图片资源
  • DataCacheGenerator从data中查找原生的未被修改的资源
  • SourceGenerator从源头找,默认策略是先缓存到磁盘，然后从磁盘加载

ResourceCacheGenerator

本文主要关心cacheFile = helper.getDiskCache().get(currentKey);

使用默认配置的话:

directory = /data/user/0/com.example.dingopractice/cache/image_manager_disk_cache

maxSize = 250 * 1024 * 1024

DiskLruCache

DiskLruCache 会读日志文件上的记录，根据该日志文件的记录将所有历史操作“重做”一遍：在执行历史操作时，会根据日志记录对缓存项进行添加和删除操作。

几个关键变量

  • directory 缓存地址
  • journalFile 日志文件
  • journalWriter 负责journalFile写入
  • maxSize最大容量
  • size 当前缓存的总容量
  • lruEntries 内存中对应实体

DiskLruCache # Entry

Entry同样有cleanFiles和dirtyFiles，写入的时候通过Editor写入

DiskLruCache 核心思想就是利用 LinkedHashMap 来做到 LRU，然后每个 Entry 中做到读写分离，互不影响。

DataCacheGenerator

与ResourceCacheGenerator相类似，不再赘述

SourceGenerator

put

MemoryCache put

由两处调用。一处为ActiveResources引用队列的清理。一处为EngineResource调用release到没有引用

DiskCache put

一处是在SourceGenerator调用cacheData

BitmapPool && ArrayPool

这两者的实现分别为LruBitmapPool和LruArrayPool。这两者并不是缓存，而是为了减少gc而使用的对象池
