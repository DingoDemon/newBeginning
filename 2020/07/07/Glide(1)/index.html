Glide源码阅读笔记(一) - url加载到ImageView全流程

发表于 2020-07-07

Glide 版本 4.11

本文梳理最基本的从网络加载一张图片到ImageView的整个流程

因篇幅原因，删略部分log代码

测试代码为
Glide.with(MainActivity.this).load("http://goo.gl/gEgYUd").into(imageView);

构造方法:

RequestManagerRetriever是创建RequestManager静态方法的集合。而RequestManager是Glide发起请求的起点。

RequestManager作为起点，Glide将其赋予了感知当前Context生命周期的能力。下文会详细说明

RequestManagerRetriever类中通过get方法来获取RequestManager。
并且重载入了数个get方法，可以通过Context，Activity，View等获取。这里以常见的Activity为例

  • getSupportRequestManagerFragment 创建了一个SupportRequestManagerFragment。
  • 在SupportRequestManagerFragment构造方法中，会创建一个ActivityFragmentLifecycle。
  • 使之LifeCircle状态为onstart,并且将其加入Activity。之后，创建RequestManager，RequestManager会将自己与Fragment的GlideLifecycle建立联系。之后RequestManager就会得到Activity生命周期的回调。

SupportRequestManagerFragment 生命周期回调

SupportRequestManagerFragment内部，回调自己生命周期的时候，会回调lifecycle的生命周期

LifeCircle 生命周期回调

在LifeCircle中会逐个回调lifecycleListener中的生命周期

RequestManager初始化

在RequestManager的构造方法内
将自身加入到LifecycleListener中

可以总结出，将SupportRequestManagerFragment加到Activity上。如果Activity的生命周期发生改变，势必会回调Fragment的生命周期,所以ActivityFragmentLifecycle会收到回调。Lifecycle回调listeners，其中就会回调到RequestManager之中。

在onDestroy中，会取消任务。也就是页面销毁了,View也就没了,图片也就没有加载的必要了。

除此之外,同样被加入的还有connectivityMonitor,这是一个监听网络的监视器。当网络情况发生变化时，reload

方法都在注释里了。接下来来看为什么是SingleRequest。参数多，代码大，简述流程

buildRequest 会调用到 buildRequestRecursive,因为测试代码没有设置errorBuilder，所以调用buildThumbnailRequestRecursive(否则返回ErrorRequestCoordinator)。

又因为没有设置缩略图(否则返回ThumbnailRequestCoordinator)所以进而调到obtainRequest最后返回SingleRequest。

所以Glide允许生成多种request

  • ErrorRequestCoordinator 正常的请求出错时,可重试
  • ThumbnailRequestCoordinator 小图请求，可以较快显示
  • SingleRequest 单次请求

A class that manages a load by adding and removing callbacks for for the load and notifying

SourceGenerator # onDataReadyInternal

Glide默认开启网络图片磁盘缓存。跟踪后续代码磁盘缓存也是使用的DiskLruCache。所以接下来以使用缓存来看

如果启用缓存

又会进入Decode的run方法，但这次runReason为SWITCH_TO_SOURCE_SERVICE

所以直接执行runGenerators方法。再次回到SourceGenerator中的startNext方法中。这次dataToCache不为空了，进行缓存

cacheData将缓存写入磁盘，然后将sourceCacheGenerator转化为DataCacheGenerator。构造DataCacheGenerator时，指定了FetcherReadyCallback为自己，所以DataCacheGenerator加载结果会由SourceGenerator转发给DecodeJob。执行他的startNext方法

DataCacheGenerator # startNext

通过这两个DataFetcherGenerator的源码，可以总结出DataFetcherGenerator一套加载流程

  1. 按照ResourceCacheGenerator，DataCacheGenerator,SourceGenerator的顺序，尝试加载model
  2. 遍历DataFetcherGenerator的modelLoaders。寻找到可以加载model的ModelLoader
  3. 可以加载model的ModelLoader有可能有数个，找到其中合法的,组装成LoadData
  4. 使用LoadData中的DataFetcher,返回数据onDataReady或者onLoadFailed
  5. onDataReady或者onLoadFailed回调到DataFetcherGenerator,最终回调到DecodeJob中的onDataFetcherReady告诉外部界面已经准备OK

再精简，就是尝试从内存，磁盘，网络来找合适的ModelLoader来加载不同的类型。如果是网络上的，会将其缓存到磁盘，并且从磁盘再加载。与我们平时所普遍认可的三级缓存相吻合。

全文流程总结

  1. Glide先调用With方法，返回一个RequestManager。这个RequestManager通过在Activity上增加Fragment的方式，感知Activity的生命周期
  2. load方法将RequestManager和model组装成为RequestBuilder
  3. into方法中，先组装一个Request，然后让RequestManager运行request，request # begin
  4. 在begin中，获取View的尺寸，如果View还未测量，则等待onPreDraw回调
  5. 在size准备好之后，使用engine # load加载
  6. engine会先从内存中查找，如果没找到的话。使用EngineJob来加载，如果已经有相同的EngineJob则按照规则复用。否则创建一个
  7. EngineJob使用线程池，在子线程中让DecodeJob来真正执行资源获取
  8. DecodeJob的run方法根据Status按照ResourceCacheGenerator，DataCacheGenerator,SourceGenerator的顺序，尝试加载model
  9. 遍历DataFetcherGenerator的modelLoaders。寻找到可以加载model的ModelLoader
  10. 可以加载model的ModelLoader有可能有数个，找到其中合法的,组装成LoadData
  11. 使用LoadData中的DataFetcher,返回数据onDataReady或者onLoadFailed
  12. onDataReady或者onLoadFailed回调到DataFetcherGenerator,最终回调到DecodeJob中的onDataFetcherReady告诉外部界面已经准备OK
  13. DecodeJob将原数据转码成需要的数据后回调EngineJob
  14. EngineJob收到回调，最终回调到Request中
  15. Request中如果有listener的话交与其处理，否则回调ViewTarget
  16. ViewTarget展示View
