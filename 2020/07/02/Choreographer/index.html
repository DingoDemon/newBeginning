Android 基于 Choreographer 的渲染机制详解

Android 怎么就不卡了呢之Choreographer

60帧的由来和垂直同步

60hz和60帧的区别

hz指的是屏幕刷新的频率，比如你屏幕刷新率是 75 hz，软件是 60 fps，每秒软件渲染60次，你刷新 75 次，是没有啥效果的，除了重复帧率费电。

帧数是指运行时每秒所运行的帧数(简称FPS,Frames Per Second)，帧数越大，画面越流畅，但是对资源消耗也就越大

引入 Vsync 之前的 Android 版本，渲染一帧相关的 Message ，中间是没有间隔的，上一帧绘制完，下一帧的 Message 紧接着就开始被处理。这样的问题就是，帧率不稳定，可能高也可能低

维持帧数围绕三部分展开，VSYNC，Triple Buffer，Choreographer

VSYNC，Triple Buffer

界面的显示大体会经过CPU的计算-> GPU合成栅格化->显示设备显示三个流程

VSYNC 每16.67ms发送一次,轮番展示画面ABC

之所以使用三级缓存的原因为最大化利用CPU，GPU。

例如:

  • 第一次VSYNC，屏幕展示画面A，CPU处理画面B，GPU等待CPU处理完毕

  • 第二次VSYNC，由于GPU在在处理画面B，所以画面停留在画面A，产生丢帧。但这时，CPU已经开始处理画面C了

  • 第三次VSYNC，GPU处理完了画面B，显示画面B，CPU处理画面A，GPU处理刚才第二次VSYNC中CPU处理的画面C。甚至在之后还能完成画面C继续处理画面A。

  • 第四次VSYNC，展示画面C。CPU/GPU处理画面B，画面A已经早早准备好了，等待下次信号

而使用两级缓存，显示器不停替换画面1和画面2的话。如果CPU或GPU之一处理超时。势必造成另一方的空闲等待。

Choreographer如何让帧数尽可能稳定在60帧

在使用垂直同步的情况下，无论多快的变化UI，刷新频率也不会小于16ms

重绘请求的提交

当UI需要展示新内容的时候，通过invalidate(重绘自身)和requestLayout(重新布局)来进行重绘。 这两者的过程都是打标记自己脏了，告诉父布局需要重绘和重新layout

最终都会经过DecorView来到ViewRootImpl中

在scheduleTraversals中 使用 Choreographer来安排下次重绘的时间

初始化主要做了这几步

  • 初始化 FrameHandler ，绑定 Looper，一般主线程的Lopper
  • 初始化FrameDisplayEventReceiver,有三个重要方法
    • onVsync – Vsync 信号回调
    • run – 执行 doFrame
    • scheduleVsync – 请求 Vsync 信号
  • 初始化CallbackQueue数组，数组内CallbackQueue对象持有CallbackRecord对象。CallbackRecord是一个链表。记录 Runnable or FrameCallback

postCallback

postCallback 方法最终会调用到postCallbackDelayedInternal方法

无论是延迟还是到期，最终都会调用scheduleFrameLocked方法

使用垂直同步

FrameDisplayEventReceiver # scheduleVsync

当 SurfaceFlinger 收到 Vsync 事件之后，通过 appEventThread 将这个事件通过 BitTube 传给 DisplayEventDispatcher ，DisplayEventDispatcher 通过 BitTube 的接收端监听到 Vsync 事件之后，回调 Choreographer.FrameDisplayEventReceiver.onVsync

this是FrameDisplayEventReceiver的run方法

doFrame

  • 做时间计算，计算一共跳过多少帧
  • 修正时间，如果时间奇怪，安排下一次垂直同步
  • do callbacks

联系上文，可以看出，安排下一帧之后，mFrameScheduled = true。如果来了新的scheduleFrameLocked不会被执行。除非doFrame完成，将mFrameScheduled置为false

callback

  • 在 CallbackQueue[] mCallbackQueues在取特定类型（输入，动画，布局，Commit）的单向链表
  • 然后取出已到期的Callback/Runable执行

几种Callbacks

Input

批量事件的处理。

Down/Up/PointerDown/PointerUP都是直接发送的，并不依赖vsync。

Move是batched input event的处理依赖于vsync。

对于move事件就有差别了。因为触摸移动中的事件不一定要每一个都处理，因为显示也就60HZ，你如果100HZ的输入事件，全处理只会浪费计算资源。对于move事件，系统会把一个VSync周期内的事件存为Batch，当VSync到来时一起处理。当VSync到来时一起处理时，会把batch的move事件合成到一个MotionEvent中，其中MotionEvent的history就是VSync内的batch 的move事件。

public void postOnAnimation(Runnable action) {
       final AttachInfo attachInfo = mAttachInfo;
       if (attachInfo != null) {

           // Postpone the runnable until we know
