View工作流程(一)

发表于 2020-05-28 | 分类于 Android

View的三大流程概述

  • Measure
  • Layout
  • Draw

一个View经过测量，布局，绘制，三个流程之后才能展示在屏幕上。

  1. View从根布局开始,测量每一个子View，每一个子View又递归测量自身。并且把测量的值储存下来。
  2. 接下来，根布局对每一个子View进行布局，安排其摆放位置。子View将其储存起来，子Viw再递归对子View进行布局。
  3. 绘制自身，绘制子View，最终，将View展示在屏幕上。

整个过程由ViewRootImpl进行调度即（performTraversals()）。在此之前，需要先搞清楚根布局,ViewRootImpl是创建，并且和Activity 关联起来的。

Measure

根据父布局的尺寸，和自身的LayoutParams。来得到子View的测量规格

measureSpec 是一个Int值，前两位表示Mode，后30位表示Dimension，Mode
有三种:

  • UNSPECIFIED 想要多大用多大，不做限制，用于测量和特殊情况(ScrollView未验证)
  • AT_MOST 父布局制定了一个最大值，最大不能超过后30位的Dimension
  • EXACTLY 精确尺寸

DecorView 产生childWidthMeasureSpec，childHeightMeasureSpec的逻辑就很明确了，MATCH_PARENT和固定尺寸是EXACTLY，WRAP_CONTENT则为AT_MOST

onMeasure

measure 方法是提供给外部调用的，View测量自身大小并告诉父布局的方法需要重写onMeasure,View对于自身的测量Android提供了默认实现，而对于ViewGroup来讲，子view怎么摆放怎么测量各不相同，对于ViewGroup类Android也必然没有默认实现

重写onMeasre

根据父控件传过来的measureSpec得到Mode，和size。根据Mode不同来进行测量:

  • EXACTLY 表示父View对⼦View 的尺⼨寸做出了精确限制，所以就放弃计算出的 size，直接选⽤用 measureSpec 的 size;

  • AT_MOST，表示父View 对⼦View 的尺⼨限制了了上限

    • 如果根据内容计算出的size不⼤于spec中限制的size，表示尺⼨寸没有超出限制，所以选⽤用计算出的 size
    • 计算出的size大于spec中限制的size，表示尺寸超限了，所以选⽤spec的size，并且如果是调用了resolveSizeAndState()的花，在其中会添加标志 MEASURED_STATE_TOO_SMALL

  • UNSPECIFIED 直接用内容计算出的size

ViewGroup里的onMeasure

策略是遍历所有子View，计算出其尺寸。并根据自身需求，计算出自身的尺寸

给子View的MeasureSpec的Mode，是由

public static int getChildMeasureSpec(int spec, int padding, int childDimension)

方法得到。

Mode+size EXACTLY AT_MOST UNSPECIFIED
dp/px EXACTLY+childSize EXACTLY+childSize EXACTLY+childSize
match_parent EXACTLY+parentSize EXACTLY+parentSize UNSPECIFIED+0
wrap_content AT_MOST+parentSize AT_MOST+parentSize UNSPECIFIED+0

在实际过程中，调用measureChild就会完成对子View 的测量。其原理就是通过父View的Mode和指定空间，调用子View的measure方法
