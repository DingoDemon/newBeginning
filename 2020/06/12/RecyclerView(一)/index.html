RecyclerView 源码分析(一) - RecyclerView的三大流程

整体简述

RecyclerView继承自ViewGroup并非AdapterView。用于灵活地展示数据集。

RecyclerView使用Adapter完成数据对子View的转换，使用LayoutManager完成子View的测量和布局，使用Recycler对划出屏幕的View进行缓存,使用ItemDecoration完成对子View装饰的处理，使用ItemAnimator来处理动画。解耦各个部分。

测量流程

既然是ViewGroup那么流程一定是根据自身mode和剩余空间测量所有子View,按照设计规则进行求总。最后得到自身尺寸。
那么从onMeasure 开始

RecyclerView # onMeasure

RecyclerView对子View的测量实际上是交给了LayoutManager进行。

all basic source

LayoutManager为空

mWidth/mHeight == null ? mWidth/mHeight : 0

LayoutManager不为空开启且自动测量

自动测量即isAutoMeasureEnabled方法返回值

系统提供提供的LinearLayoutManager 和GridLayoutManager都return true。所以使用这两种LayoutManager 会自动测量。

LinearLayoutManager # isAutoMeasureEnabled

RecyclerView # onMeaseure (part1)

mLayout.onMeasure(mRecycler, mState, widthSpec, heightSpec);
mLayout.onMeasure让人迷惑。

但其实传统的LayoutManager都没有实现这个方法，执行的是mRecyclerView.defaultOnMeasure(widthSpec, heightSpec);这也是考废弃方法，忽略。

measureSpecModeIsExactly

如果长宽的mode都为精确尺寸，或者没有设置adapter，那么直接return。这时长宽为defaultOnMeasure

State && dispatchLayoutStep

RecyclerView.State 这个类封装了当前RecyclerView的有用信息。

State的一个变量mLayoutStep表示了RecyclerView当前的布局状态，包括STEP_START、STEP_LAYOUT 、 STEP_ANIMATIONS三个。

而RecyclerView的布局过程也分为三步，其中，STEP_START表示即将开始布局，需要调用dispatchLayoutStep1来执行第一步布局，布局状态变为STEP_LAYOUT。

接下来需要调用dispatchLayoutStep2里进行第二步布局，同理，第二步布局后状态变为STEP_ANIMATIONS。

最后执行第三步布局dispatchLayoutStep3。

根据State状态，如果OnMeasure如果已经进行了第1，2步，那么后续dispatchLayout方法里只会执行第三步，避免重复测量

if (mState.mLayoutStep == RecyclerView.State.STEP_START) {dispatchLayoutStep1();}

dispatchLayoutStep1，2，3

method describe
dispatchLayoutStep1 处理Adapter更新，决定哪种动画应该执行,保存现有View状态，如果需要的话，处理预布局
dispatchLayoutStep2 真正执行子view布局，有可能会执行多次
dispatchLayoutStep3 执行动画逻辑，将状态重新置为STEP_START

RecyclerView # dispatchLayoutStep1

  • 根据动画不同，为当前显示在界面上的每一个 ViewHolder 创建一个 ItemHolderInfo，储存在ViewInfoStore中。ItemHolderInfo 其实就是保存了当前显示 itemview 的布局的 top、left 等信息。
  • 将mState.mLayoutStep置为STEP_LAYOUT

紧接着，进入dispatchLayoutStep2

dispatchLayoutStep2

LinearLayoutManager

以LinearLayoutManager的代码来看

LinearLayoutManager # onLayoutChildren

layout algorithm:

  1. by checking children and other variables, find an anchor coordinate and an anchor item position.
  2. fill towards start, stacking from bottom
  3. fill towards end, stacking from top
  4. scroll to fulfill requirements like stack from bottom.
    create layout state

LinearLayoutManager # updateAnchorInfoForLayout

mAnchorInfo为布局锚点信息，包含了子控件在Y轴上起始绘制偏移量（coordinate），ItemView在Adapter中的索引位置（position）和布局方向（mLayoutFromEnd）——这里是指start、end方向。这部分代码的功能就是：确定布局锚点，以此为起点向开始和结束方向填充ItemView

LinearLayoutManager # fill

在onLayoutChildren中从上到下从下到上fill两次的原因在于

起点是锚点。RecyclerView第一次测量的时候，当然是从0开始。但是在运行过程中，scroll到某个postion或者中部View获取焦点。那么起点就不是顶点。就需要从中部从上到下，从下到上开始测量

LinearLayoutManager # layoutChunk

RecyclerView # measureChildWithMargins
RecyclerView # getChildMeasureSpec

关系到获取子View的MeasureSpec，是个值得一看的方法

int getChildMeasureSpec(int parentSize, int parentMode, int padding, int childDimension, boolean canScroll)

params describe supplement
parentSize Size of the parent view where the child will be placed 一般情况下，走到这里的时候。recycler的父布局早已为它安排好了大小。
parentMode The measurement spec mode of the parent. 无
padding Total space currently consumed by other elements of parent RecyclerView 的 padding
childDimension Desired size of the child view, or MATCH_PARENT/WRAP_CONTENT. Generally obtained from the child view’s LayoutParams MATCH_PARENT = -1,WRAP_CONTENT = -2
canScroll true if the parent RecyclerView can scroll in this dimension canScrollHorizontally/canScrollHorizontally ,如果布局是horizontal的话可以横滑，vertical同理
LinearLayoutManager # layoutDecoratedWithMargins

为子View安排layout，至此每个子View的尺寸位置确定。layoutChunk 结束

fill两次后,除了一些后续处理。onLayoutChildren方法结束。可以回到RecyclerView的onMeasure中
