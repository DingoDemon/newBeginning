RecyclerView(二) - 缓存

发表于 2020-06-14

鸣谢:

深入理解 RecyclerView 的缓存机制

RecyclerView预加载机制源码分析

概述

RecyclerView 在滑动中不能每次都将移除的View销毁，创建新的添加进来。也不能无限地添加不销毁。RecyclerView内部实现了三级缓存，使之能够:

  • 上下滚动时利用复用机制，往下滑动时将移去的View缓存下来。往回滑动时如果可复用，移入时从缓存里添加回来。

  • 在向一个方向滑动时，计算即将滑入的View，并在规定时间内，尝试将其加入缓存。更高效地利用UIThread

  • 在全局刷新(NotifyDataSetChange)的时候，亦有缓存的应用,将还可复用的item，再加回来。

  • 甚至在多个RecyclerView之间，如果是相同的Adapter&&ViewType，那么无疑需要使用缓存。

Recycler

RecyclerView 抽离 Recycler来进行缓存的处理

A Recycler is responsible for managing scrapped or detached item views for reuse.

A “scrapped” view is a view that is still attached to its parent RecyclerView but that has been marked for removal or reuse

Typical use of a Recycler by a LayoutManager will be to obtain views for an adapter’s data set representing the data at a given position or item ID.
If the view to be reused is considered “dirty” the adapter will be asked to rebind it.
If not, the view can be quickly reused by the LayoutManager with no further work.

缓存有存和取两个流程组成。先看取的过程，再看存的过程。

一级缓存

ArrayList<ViewHolder> mChangedScrap

ArrayList<ViewHolder> mAttachedScrap

ArrayList<ViewHolder> mCachedViews

  • mChangedScrap用于动画时的内部缓存。

  • mAttachedScrap 用于数据刷新时(notify)，储存被detach的View。调用处为onLayoutChildren # detachAndScrapAttachedViews(recycler)

mChangedScrap和mAttachedScrap 只会在布局阶段不为空。

  • mCachedViews用于缓存屏幕滑入滑出时进行的缓存,默认最大缓存2个。

二级缓存

ViewCacheExtension mViewCacheExtension;

用户自定义缓存，需实现

View getViewForPositionAndType(@NonNull Recycler recycler, int position, int type)方法。

三级缓存

缓存量更大的缓存，实现一二级缓存缓存不下的缓存，和多个RecyclerView之间共享缓存

  • RecycledViewPool mRecyclerPool;

RecycledViewPool lets you share Views between multiple RecyclerViews

布局时，尝试一/二/三级缓存 中 获取 ViewHolder

即取的过程

获取ViewHolder的流程为:

  • 一级缓存中没有，则去二级缓存中查找，二级中没有，去三级中查找。如果还是没有，则创建一个。

这个过程，来自于测量流程里的 LayoutManager # fill # layoutChunk。这个方法会尝试铺满整个布局。这个时候就会执行获取ViewHolder的流程。即 RecyclerView # tryGetViewHolderForPositionByDeadline

滑动过程中的复用

存的部分，之一

RecyclerView的滑动过程可以分为2个阶段：手指在屏幕上移动，使RecyclerView滑动的过程，可以称为scroll；手指离开屏幕，RecyclerView继续滑动一段距离的过程，可以称为fling

滑动:

  • ACTION_MOVE
  • ACTION_UP

ACTION_MOVE

ACTION_MOVE 会把偏移量，Recycler，和 State 传递给LayoutManager进行处理。毕竟measure，layout都在LayoutManager中进行。还是以LinearLayoutManager来举例。

fill 方法 的返回值变得重要起来

@return Number of pixels that it added. Useful for scroll functions.

回到fill方法中。

不管是从头还是从尾开始，都会来到recycleChildren,并最后进入removeAndRecycleViewAt方法

将即将划出屏幕的View进行回收

之后调用`recycleViewHolderInternal(holder)’

mCachedViews遵循先进先出的原则进行缓存。当超过缓存的时候，会向pool进行缓存,此时被移除到RecyclePool的ViewHolder的内容会被标记为无效，当其再次使用时需要再次通过Adapter.bindViewHolder来绑定内容。

预加载

当滑动的时候会预加载即将滑入的View，如果在时间内能加载成功，则进入缓存，滑到正式进入的时候，从mCachedViews中直接取

notifyDataSetChange时的缓存处理

存的部分，之二

notifyDataSetChange的时候，RecyclerView不知道哪些View变了，那些View没变。会调用ViewGroup的requestLayout整体重新布局。

可以看到所有View都与RecyclerView相剥。最后缓存进RecyclerPool中的mScrap内。

并在之后fill的时候从缓存中获取


public ViewHolder getRecycledView(int viewType) {
          final ScrapData scrapData = mScrap.get(viewType);
          if (scrapData != null && !scrapData.mScrapHeap.isEmpty()) {
              final ArrayList<ViewHolder> scrapHeap = scrapData.mScrapHeap;
              return scrapHeap.remove(scrapHeap.size() - 1);
          }
          return null;
      }
